Exploring tus.io

April 27th, 2017 by Ameya Advankar

Problem Statement:

Scientists utilizing the Scientific Gateways often upload large input files for running Experiments. These uploads which are done through the Science Gateway Portals are over HTTP. There are 2 inherent problems arising out of this architecture:
  1. Unreliable HTTP connection:

    As the uploads are over HTTP, these are highly reliant on a continuous internet connection being available on the Client machine. Often times, there are network disruptions or connectivity issues and the File upload fails. As a result, the users may have to retry uploads manually and wait for a successful upload to take place.

  2. Space constraints on Gateway servers:

    Since the file is being uploaded to the Science Gateway portal, it has to be staged in the Portal until it is required which may be a considerable amount of time. In case multiple large files are uploaded at the same time, the web server running the Science Gateway may run out of server space and this could have adverse affects on the performance.

    Also, Many a times, multiple Science Gateway Portals are hosted on a Single Web server. In this scenario, a particular Portal which gets multiple large files may affect other Portals performance as well.

Problem Evaluation:

To combat the First issue of Unreliable HTTP connection, it has to be ensured that subsequent auto-retries for uploading the file are done. Along with auto-retries, the data transfer between Client and server should be minimized by not re-uploading chunks of data which was already uploaded prior to failure of connection. This feature has been implemented by multiple Javascript libraries. Notable among these are :

Resumable.js Flow.js File Uploader tus.io Client/Server

All the Javascript libraries mentioned above are fully capable of performing resumable uploads. However due to their unique implementations, it may be difficult to extend the functionalities to suit our applications need.

tus.io establishes an open protocol which aims to solve the problem of unreliable file uploads once and for all. To think of it, tus.io specifies a set of rules, which, if followed by the client & server implementations, would allow any such implementation to successfully exhibit reliable and resumable File upload system.

The second problem pertaining to space issues arising out of uploaded content, can be solved by having the upload Server-implementation run on a dedicated separate remote Application server instance. This way, in case of large upload files, the upload traffic gets directed to the upload server and the Application Server on which the Scientific Portals are located would not get hampered.

These remote Server instances which would be used for Uploading files, should be secured in a similar manner as the Portals are secured i.e. using an Identity and Access Management system. tus.io has a concept of hooks which allows us to execute code at certain stages of the upload proces. The pre-create hook can be used for implementing Auth checking in the upload server. This prevents any unauthorized access to the File-Upload server and thus ensures the security of the entire system.
